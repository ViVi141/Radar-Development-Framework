# 雷达模块 — 计划与需求

本文档描述雷达模块的**计划需求**：在 RDF 中（重新）引入雷达能力，重点包括**炮弹/子弹的扫描与追踪**，以及**雷达本身可被其他雷达探测**（主动雷达与被动雷达概念）。当前仓库仅保留 LiDAR；雷达为规划中功能，实现时需新增代码与数据。

---

## 一、目标概述

| 需求项 | 说明 |
|--------|------|
| **炮弹/子弹扫描与追踪** | 雷达能够检测、识别并持续追踪飞行中的炮弹与子弹（抛射物实体），输出位置、速度及轨迹。 |
| **雷达可被探测** | 雷达站/载具上的雷达作为**可探测目标**：主动工作时可被对方雷达发现（发射特征），被动或静默时探测难度更高，形成主动/被动雷达对抗。 |

---

## 二、炮弹/子弹的扫描与追踪

### 2.1 功能需求

- **检测**：在雷达扫描范围内，识别并列出所有“抛射物”类实体（炮弹、子弹、火箭弹、导弹等），不限于载具与步兵。
- **追踪**：对已检测到的抛射物进行多帧关联，形成轨迹（位置随时间、可选速度矢量），支持：
  - 单目标轨迹记录（最近 N 帧位置/速度）；
  - 多目标同时追踪（按实体 ID 或运动学关联去重）。
- **输出**：与现有“目标”抽象一致，便于接入 HUD、告警与武器系统：
  - 位置、距离、方位/俯仰；
  - 速度矢量（若引擎提供或可差分估计）；
  - 目标类型/分类（炮弹/子弹/导弹等，若可区分）。

### 2.2 实现思路（与现有架构兼容）

- **实体来源**：沿用“实体优先”思路（见 [VEHICLE_RADAR_LOCK_GUIDE.md](VEHICLE_RADAR_LOCK_GUIDE.md) 第五节）：
  - 使用引擎 `QueryEntitiesBySphere` / `QueryEntitiesByAABB` 等在扫描半径内获取候选实体；
  - **过滤**：增加对**抛射物**的识别（例如实体类型、父级、组件或脚本标签：Reforger 中与弹体/抛射物相关的实体或组件）；
  - 对候选做射线判可见（从雷达原点到弹体位置或 AABB 代表点），避免被地形/建筑遮挡仍报目标。
- **追踪层**：新增**抛射物追踪器**（或扩展现有锁定/追踪模块）：
  - 输入：本帧检测到的抛射物列表（实体 + 位置 + 可选速度）；
  - 跨帧关联：按实体句柄或位置/速度预测做关联，维持“同一发弹”的轨迹；
  - 输出：当前所有被追踪的抛射物及其轨迹（最近 N 帧），供 HUD、告警、近防或反击逻辑使用。
- **与 LiDAR 的关系**：若雷达实现采用“实体优先 + 射线判可见”，可与 LiDAR 共用射线与可见性逻辑；仅实体过滤与目标分类不同（载具 vs 抛射物）。若后续雷达采用电磁波/波束模型，则扫描与追踪接口可统一为“目标列表 + 轨迹”，便于复用 HUD 与告警。

### 2.3 依赖与约束

- **引擎**：需确认 Reforger 中抛射物/子弹的实体类型、组件或查询方式（如是否可通过 `QueryEntities*` 得到、是否有“Projectile”或类似分类）。
- **性能**：抛射物数量可能较大（多发齐射、弹幕），需对“同时追踪数量”和“轨迹长度”设上限，并做空间或时间降采样，见 [OPTIMIZATION_AND_MEMORY.md](OPTIMIZATION_AND_MEMORY.md)。

## 三、雷达本身可被其他雷达探测（主动与被动）

### 3.1 概念

- **主动雷达**：发射电磁波（或在本框架中抽象为“发射信号”），通过回波检测目标。  
  - **可被探测性**：发射时会产生**辐射信号**，可被对方被动接收机或主动雷达侧向发现，即“雷达作为目标”可被探测。
- **被动雷达**：仅接收信号（或本机不主动发射），依赖对方辐射或第三方辐射探测目标。  
  - **可被探测性**：不发射时本机雷达难以被对方直接通过“辐射”发现，仅可能通过光学/其他传感器发现载具或阵地。

需求：**雷达实体在“主动工作”时应具有可被探测属性；在“被动/静默”时可配置为不可被探测或显著降低被探测概率。**

### 3.2 功能需求

- **辐射/发射状态**：每个雷达实体有“是否正在发射”的状态（如开机并扫描、待机静默等）。
- **可探测签名**：当雷达处于“主动发射”时，对外暴露**可被查询的签名**，例如：
  - **辐射强度**（或等效 RCS/发射功率等级）：用于对方雷达或被动传感器计算“是否可被发现、在多大距离可被发现”；
  - **位置与朝向**：雷达实体世界坐标（及可选朝向），供对方扫描时用作目标位置。
- **探测方逻辑**：对方雷达（或被动传感器）在扫描时，除载具、步兵、抛射物外，还能查询到“正在辐射的雷达”实体，并将其作为一类目标显示在 HUD、用于告警或反辐射打击。

### 3.3 实现思路

- **雷达实体注册**：所有“可被探测的雷达”在场景中注册到一个全局或世界子系统（或通过组件挂载在实体上），并上报：
  - 实体引用、世界位置、朝向；
  - 当前是否发射、辐射强度（或功率等级）。
- **探测方查询**：对方雷达扫描时，除引擎 `QueryEntities*` 得到的实体外，再查询“已注册且正在发射的雷达”列表，按距离与方位过滤后，与常规目标一起参与可见性判断（可选：对雷达目标做射线判可见或简化为“在扇区内即可见”）。
- **主动/被动模式**：  
  - **主动模式**：雷达执行扫描（发射），注册表中该实体标记为“发射中”，参与“可被探测”查询。  
  - **被动模式**：雷达仅接收、不发射，或关闭扫描；注册表中该实体标记为“静默”，不从“可被探测雷达”列表返回，或返回时带“静默”标记且探测方忽略/降权。

这样即可实现“雷达本身可被其他雷达探测”，且支持主动/被动两种工作方式。

### 3.4 与现有模块的关系

- **与 LiDAR**：LiDAR 为光学/激光，不涉及电磁辐射与“被探测”语义；本需求针对**雷达**模块。若未来雷达与 LiDAR 共存在同一载具上，可分别配置（例如雷达主动时载具可被对方雷达发现，LiDAR 不参与该逻辑）。
- **与载具锁定**：[VEHICLE_RADAR_LOCK_GUIDE.md](VEHICLE_RADAR_LOCK_GUIDE.md) 中的锁定管理器可扩展为：目标类型除“载具”外，增加“雷达”（主动辐射的雷达实体）与“抛射物”，实现反辐射锁定与炮弹告警。

## 四、实施优先级建议

| 优先级 | 内容 | 说明 |
|--------|------|------|
| P1 | 炮弹/子弹检测与列表输出 | 在扫描范围内识别抛射物实体并输出目标列表，与现有实体优先 + 射线判可见方案结合。 |
| P2 | 抛射物多帧追踪与轨迹 | 跨帧关联同一抛射物，输出轨迹（最近 N 帧位置/速度），供 HUD 与告警使用。 |
| P3 | 雷达可被探测（主动辐射注册与查询） | 雷达实体注册、发射状态、辐射强度；对方扫描时可查询到“正在发射的雷达”并作为目标。 |
| P4 | 被动/静默模式 | 雷达不发射时不参与“可被探测”列表，或降权显示。 |
