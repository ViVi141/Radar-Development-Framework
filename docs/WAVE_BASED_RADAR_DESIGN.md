# 完全波/场基雷达设计（抛弃射线检测）

本文档描述一种**不再使用任何“扫描射线”（TraceMove）**的电磁波雷达架构：目标检测完全由**体素场中的能量传播与采样**完成，几何信息来自**体素占位网格**而非逐向量的射线检测。

---

## 1. 目标与约束

- **完全抛弃**：雷达检测阶段不再调用 `TraceMove` 或任何“按方向打射线取首碰”的 API。
- **保留**：上层仍可输出 `RDF_RadarSample` 或等价结构（距离、方位、功率、可选实体引用），以便现有 HUD、CFAR、分类器复用。
- **几何来源**：仅允许“体素占位”的构建与查询（高度图 + 实体 AABB），以及**可选的**“按检测点查实体”（每个检测点一次，而非每条扫描射线一次）。

---

## 2. 整体数据流

```
[发射]  →  向体素场注入能量（方向 + 功率 + 时间戳）
              ↓
[传播]  →  每帧 TickPropagate：能量在体素间扩散，遇固体则反射，无射线
              ↓
[接收]  →  在接收机位置/扇区内采样场（功率 + 主方向 + 到达时间）
              ↓
[检测]  →  按时延 → 距离、按方向 → 方位，聚成 (距离, 方位, 功率)
              ↓
[输出]  →  转为 RDF_RadarSample（或兼容列表），可选：按检测点查实体
```

几何只参与：**(1) 体素占位网格的填充**、**(2) 传播时“是否为固体”判断**、**(3) 可选“检测点→实体”解析**。不再有“每条扫描方向一条射线”。

---

## 3. 体素占位层（零射线几何）

### 3.1 用途

- 传播时判断：从格子 A 到邻居 B，B 是否为**固体**（地形/建筑/实体）；若为固体，则不在 B 累积能量，而是生成反射到相邻空气格。
- 可选：在固体格记录“代表实体”（用于检测阶段反查“命中的是谁”）。

### 3.2 与现有 EM 网格对齐

- 使用与 `EMVoxelField` 相同的**块/格划分**：宏块 200 m，微格 10 m（20³ = 8000 格/块），以便传播与占位一一对应。
- 每个 chunk 增加（或独立模块提供）：
  - **占位数组**：`m_Occupancy[8000]`，取值如 0=空、1=地形、2=实体等；
  - 可选 **m_EntityIndex[8000]**：若为实体格，存实体 ID 或引用，便于“检测点→实体”解析。

### 3.3 填充方式（无扫描射线）

- **地形**：对每个微格中心 `(x, y, z)`，调用 `GetSurfaceY(x, z)`（或引擎等价高度查询）。若 `y <= surfaceY + 阈值`，则该格标为地形。仅用高度查询，无射线。
- **实体**：对当前 chunk 的 200 m 包围盒做 `QueryEntitiesInBounds`（或等效），得到与该块相交的实体列表；对每个实体取世界 AABB，对与 AABB 相交的微格中心做“点是否在 AABB 内”判断，若在则标为实体并记录该实体。无射线。

占位可在以下时机更新：

- 块首次被传播/注入涉及到时（懒加载）；
- 或每 N 秒/每帧仅更新少数块（轮转），以控制 CPU。

---

## 4. 发射（无射线）

- 发射机**只向体素场写入**，不读回、不 Trace。
- 沿用现有接口即可：
  - `InjectAlongRay(origin, dir, range, power, ...)`：沿方向逐步写格，仍是“沿一条直线写场”，不是“射线检测命中谁”。
  - 或 `PushRay(...)`：惰性射线，在采样时按 1/R² 等贡献功率。
- 需为每次发射（或每个脉冲）打上**发射时间戳**，以便接收端用“到达时间 − 发射时间”得到双程时延 → 距离。

可选：在写入时同时写入“发射时间”或“脉冲 ID”到格子的元数据（若当前 EMChunk 无此字段，可在传播/接收层用单独结构按 (chunk, cell) 记录最近一次脉冲时间）。

---

## 5. 传播（波在体素间扩散，无射线）

### 5.1 TickPropagate 概念

- 每帧（或每若干帧）对**有能量的 chunk** 执行一次传播步：
  - 对每个有功率的微格，向 **6 邻**（或 26 邻）扩散一部分能量；
  - 扩散系数：按距离衰减（如 1/距离² 或常数因子），并乘以 `dt` 与传播速度的换算。
- **固体格**：若邻居为占位=固体，则不向该格写入，改为**反射**：
  - 根据固体格法线（可由占位网格的梯度或预存法线近似）计算反射方向；
  - 将本应写入固体格的那部分能量，写入“反射方向上的相邻空气格”（或沿反射方向再走一步的格）。

这样，波前在体素间一步步推进，遇到地形/实体就反射，**全程不调用 TraceMove**。

### 5.2 反射与衰减

- 反射系数：可依“实体/地形”或未来扩展的材质表取不同值（0~1）；
- 传播衰减：每步按自由空间损耗或固定衰减率缩小，与现有 `TickDecay` 可合并或分步。

### 5.3 性能控制

- 只对**有能量且在有源扇区内的 chunk** 做传播；
- 每帧可限制传播的 chunk 数或格子数（分片传播）；
- 可选：传播步长与 EM 的 LoD 一致（如近处每帧传，远处每 N 帧传一次）。

---

## 6. 接收与检测（从场到目标列表）

### 6.1 在接收机处采样场

- 接收机位置 = 雷达天线位置（与发射同址或已知偏移）。
- 每个“检测周期”内：
  - 在**当前扇区**内，对体素场做采样：可用现有 `GetPowerAt(pos, outDir, outFreq)`，并扩展为带**到达时间**或**脉冲 ID**（若已写入）；
  - 或：对扇区内格子做批量遍历，读取功率 + 主方向 + 时间戳。

### 6.2 从“功率–时间–方向”到目标

- **距离**：`range = (t_receive - t_transmit) * c / 2`（双程时延）。若场里未存时间戳，可退化为“当前帧接收功率”对应“上一脉冲的某距离环”，由脉冲宽度与 PRF 近似。
- **方位/俯仰**：由格子的主方向或接收波束权重重建。
- **功率**：直接来自场采样，可再换算为 SNR/dB（相对噪声底）。

将 (距离, 方位, 俯仰, 功率) 做**距离–角度二维聚类**（或简单 binning），得到若干“检测点”，每个对应一个目标候选。

### 6.3 实体身份（可选，最小化射线）

- **方案 A（零射线）**：占位网格在实体格已记录“代表实体”；检测点落在某格时，直接读该格的实体引用作为 `hitEntity`。
- **方案 B（每检测点一次查询）**：仅对**最终聚类后的检测点**（数量远小于射线数），在检测点位置做一次“点/小盒重叠查询”得到实体。仍可视为“不依赖扫描射线”，因为射线数从 O(扫描方向) 降为 O(检测目标数)。

---

## 7. 与现有组件的对接

- **RDF_RadarSample**：由每个“检测点”填充一条 sample：`m_Distance`= 距离，`m_HitPos`= 检测点位置，`m_ReceivedPower`/SNR 来自场功率，`m_Entity`= 上节的实体解析（可选），`m_Dir`= 方向。其他字段（多普勒、RCS 等）可由距离/速度/方向事后估算或留空。
- **RDF_RadarScanner**：新增“波模式”分支：不调用 `Scan()` 中的射线循环，改为调用“场发射 → TickPropagate（若本帧轮到此雷达）→ 场采样 → 检测聚类 → 填 outSamples”。原有 `ApplyRadarPhysics` 可对聚类结果做简化版功率/SNR 计算，或由场采样直接提供。
- **HUD / CFAR / 分类器**：继续消费 `array<ref RDF_LidarSample>`，无需改接口。

---

## 8. 实现阶段建议

| 阶段 | 内容 | 射线使用 |
|------|------|----------|
| 1 | 体素占位网格（高度图 + 实体 AABB），懒加载，只读不写 | 0 |
| 2 | TickPropagate：6 邻扩散 + 固体反射（用法线近似） | 0 |
| 3 | 发射时间戳写入与读取；接收端按时间→距离 bin | 0 |
| 4 | 从场采样生成 (距离, 方位, 功率)，聚类为检测点 | 0 |
| 5 | 输出 RDF_RadarSample，对接现有 HUD/CFAR | 0 |
| 6 | 可选：检测点→实体（占位表 或 每点一次重叠查询） | 0 或 O(检测数) |

整体可做到**雷达检测主路径零 TraceMove**；若采用“每检测点一次实体查询”，则射线量从“每扫描方向一条”降为“每目标一条”，且可再被占位表替代为 0。

---

## 9. 风险与取舍

- **性能**：传播步是 O(活跃格子数 × 邻域数)，需严格限制活跃 chunk 数量与更新频率；与现有 EM 扇区、chunk 预算一致即可。
- **精度**：体素 10 m 会限制距离/方位分辨率，可通过更细网格或子格插值缓解，代价是内存与 CPU。
- **多普勒/速度**：纯场采样不直接给出速度，需多帧检测点关联或从实体速度反推（若已解析实体）。

该设计满足“完全抛弃扫描射线”的目标，同时保留与现有雷达 API 和 HUD 的兼容性，并明确几何仅用于体素占位与可选的按点实体解析。

## 10. 完全波方案详细说明

下面按“物理直觉 → 各环节实现细节 → 数据与性能”展开。

### 10.1 物理直觉：波如何在空间里“走”

- **射线方案**：从天线沿某方向打一条线，只关心这条线与场景的**第一个交点**，得到“该方向上的目标”。中间有树枝/地形就会断，且每条方向一条线，成本随方向数线性涨。
- **波方案**：把空间切成小格（体素），每格存“当前有多少电磁能量、大致从哪来”。  
  - **发射**：天线所在格（及沿波束方向的若干格）被写入能量，并记下“这是第几脉冲、何时发出”。  
  - **传播**：每帧（或每 N 帧）做一步：有能量的格把一部分能量“分给邻居格”；若邻居是**固体**（地形/实体），不往固体里写，而是按反射方向写回**空气格**。这样能量会沿空间扩散、遇障反射，相当于波前在体素网格上推进。  
  - **接收**：在天线位置（或扇区内）**采样场**：某格在某一时刻的功率 + 主方向 + 若有的“到达时间”。功率大且时间与发射时间差 ≈ 2R/c 的，就对应距离 R 处的回波。  
- 因此：**几何遮挡**不再是一根线断就全断；波可以“绕”到旁边格再往前传（在离散格子上体现为扩散到邻格），树枝只挡掉一部分截面时，仍可能有能量从旁侧到达目标并返回，从而实现“几何连线被挡仍可能捕获”。

### 10.2 体素占位层（几何从哪来、零射线）

**作用**：告诉传播逻辑“哪格是空气、哪格是固体”，以及固体的大致朝向（法线），用于反射。

- **网格**：与现有 `EMVoxelField` 一致——宏块 200 m，微格 10 m，每块 20³ = 8000 格。每个 chunk 配一块**占位数组** `Occupancy[8000]`，取值例如：0=空、1=地形、2=实体；可选再配 `EntityRef[8000]`，实体格存对应 `IEntity` 或 ID。
- **地形填充**：对格心世界坐标 `(x, y, z)` 调 `GetSurfaceY(x, z)` 得到地表高度。若 `y <= surfaceY + 容差`（例如 0.5 m），则该格标为地形。**不做射线**，只做高度查询。
- **实体填充**：  
  - 用引擎 API 取“与当前 200 m 块 AABB 相交”的实体列表（如 `QueryEntitiesInBounds` 或等效）。  
  - 对每个实体取世界 AABB；对块内每个微格中心，若落在某实体 AABB 内，则该格标为实体并写入 `EntityRef`。  
  同样**无射线**，仅 AABB 与点包含关系。
- **法线（固体朝向）**：  
  - **地形**：可用相邻格高度差近似梯度，得到“上”方向，反射时用入射方向对该法线做镜面反射。  
  - **实体**：可用该格所在实体 AABB 的最近面法线近似，或统一用“从实体中心指向格心”的方向作为简单近似。  
  用于 10.4 的反射计算。
- **更新策略**：  
  - **懒加载**：某块第一次被发射/传播涉及到时，再填该块的占位。  
  - **轮转更新**：每帧只更新少数块（如 1～4 块），实体多的块可提高更新频率，避免单帧集中算全图。

### 10.3 发射（只写场、不检测）

- 雷达“开机”时，不调用任何 Trace，只调用现有**写场**接口：  
  - `InjectAlongRay(origin, dir, range, power, freqMask, ttl, ...)`：沿 `dir` 从 `origin` 起以固定步长（如 10 m）逐步写体素，每格写入的功率可按 1/R² 或均匀分配；或  
  - `PushRay(...)`：惰性射线，在后续“接收采样”时再按 1/R² 等贡献（若采用“传播+采样”为主，可仍用 InjectAlongRay 以便能量真的进体素参与传播）。
- **时间戳**：每次发射（每个脉冲）需要“发射时刻”`t_tx`。做法之一：在写入体素时，同时往该格的**元数据**里写 `lastPulseTime = t_tx`（若当前 EMChunk 无此字段，可单独维护一个 `map<(chunk,cell), pulseTime>` 或 per-cell 的轻量结构）。这样后续“接收”时才能用“到达时间 − 发射时间”算双程时延。
- 脉冲重复间隔（PRF）与脉宽由上层雷达模式决定；场层只关心“这一帧写入了多少能量、对应哪一脉冲、何时发出”。

### 10.4 传播（波在体素间扩散 + 反射）

- **每步做什么**：对每个“当前有能量”的微格，把其能量的一部分**向 6 邻**（上下左右前后）转移。转移量可设为：  
  `outFlow = power * diffusionRate * dt`，其中 `diffusionRate` 是 0～1 的常数，与“波速/格大小”量纲一致；或按距离衰减（如邻格共 6 个，每个拿 1/6 再乘衰减）。  
  写入**目标格**时：若目标格为**空气**，则累加；若为**固体**，则不写入该格，改为计算**反射**（见下）。
- **反射**：设当前格能量主方向为 `d_in`，固体格法线为 `n`（见 10.2）。反射方向 `d_out = d_in - 2*dot(d_in,n)*n`（镜面反射）。将本应写入固体格的那份能量，改为写入“沿 `d_out` 方向的**相邻空气格**”（即与当前格共享面且法线最接近 `d_out` 的那一格，或沿 `d_out` 一步后的格）。反射系数 `rho`（0～1）乘在反射能量上，可依地形/实体或材质区分。
- **衰减**：每步除扩散外，还可对格内能量乘 `exp(-decay*dt)` 或线性衰减，与现有 `TickDecay` 一致，表示传播损耗与时间衰减。
- **边界**：只对“在活跃扇区内”的 chunk 做传播；块数、每帧传播的块数可严格上限（如每帧最多 8 块），避免单帧爆量。

### 10.5 接收与检测（从场到“目标列表”）

- **接收位置**：与发射同址（单站雷达），即天线所在世界坐标；若有偏移则用配置的接收机偏移。
- **采样方式**：  
  - **方式 A**：在接收点调用 `GetPowerAt(receiverPos, outDir, outFreq)`，若扩展了“到达时间”，则同时得到该点当前功率、主方向、以及最近一次脉冲到达时间 `t_rx`；距离 `R = (t_rx - t_tx) * c / 2`。  
  - **方式 B**：对“当前扇区内”的体素做遍历（例如按距离环 + 方位 bin 遍历），每格读功率、主方向、时间戳，再按距离/角度聚合。  
  方式 A 适合“单点接收、多距离门”；方式 B 适合“整个扇区做距离–方位二维图”。
- **距离–方位 bin**：将 (R, azimuth, elevation) 离散成 bin，同一 bin 内的功率可相加或取 max，得到“距离–方位”二维直方图；再在直方图上做**聚类**（如局部极大值、或超过门限的连通域质心），得到若干 (R, az, el, power) 的“检测点”。
- **门限**：功率低于噪声底或某阈值的 bin 不参与聚类，减少虚警。阈值可与现有雷达检测门限（dB）对应。

### 10.6 从检测点到 RDF_RadarSample（对接现有 API）

- 每个聚类得到的检测点 → 一条 `RDF_RadarSample`（或兼容结构）：  
  - `m_Distance` = R；`m_HitPos` = 由 (R, az, el) 与接收机位置算出的世界坐标；`m_ReceivedPower` / SNR 由场功率换算；`m_Dir` = 该 bin 的主方向或从接收机指向 `m_HitPos`。  
  - `m_Entity`：若占位里该格存了实体引用，直接填；否则可选“仅对检测点做一次点/盒重叠查询”得到实体（射线数 = 检测点数，通常远小于原射线数）。
- 多普勒、RCS 等：纯场采样不直接给出；可用“多帧检测点关联 + 距离变化”估速度，或若已解析实体则用实体速度；RCS 可留空或由功率与距离反推近似。

### 10.7 性能量级与和射线方案对比

- **射线方案**：成本 ≈ 射线数 × (1 或 2) × TraceMove 单次成本 × 每秒扫描次数；射线多、扫频高或开烟雾遮挡时，Trace 量成倍涨。
- **波方案**：  
  - 占位：仅在被涉及块上做高度查询 + 实体 AABB 包含，无 Trace。  
  - 传播：每帧只处理“有能量”的 chunk，每格 6 邻读写，成本 ≈ 活跃格数 × 常数；可通过“每帧只传播部分块、限制活跃块数”控住。  
  - 接收：扇区内体素遍历或单点多次 GetPowerAt，与射线数无关。  
- 因此：**波方案把“扫描射线”从 O(射线数) 降为 0**；性能瓶颈从“大量 Trace”转为“体素传播与占位更新”的规模，二者量纲不同，在“射线很贵”的引擎里波方案更有机会稳住帧率。

### 10.8 实现顺序建议（与第 8 节一致）

1. **占位**：实现与 EM 对齐的占位网格，地形 + 实体填充，懒加载/轮转更新；可选法线。  
2. **传播**：在 EMVoxelField 上增加 `TickPropagate(dt)`，6 邻扩散 + 固体反射。  
3. **时间戳**：发射写脉冲时间，接收可读时间 → 距离。  
4. **接收与聚类**：扇区采样 → 距离–方位 bin → 聚类 → 检测点。  
5. **输出**：检测点 → RDF_RadarSample，对接 HUD/CFAR。  
6. **实体**：占位表反查或每检测点一次查询。

按上述顺序可实现**完全波方案**：几何仅用于占位与可选实体解析，检测主路径零 TraceMove，同时保留与现有雷达 API 的兼容性。

## 11. 技术评审与关键挑战对策

本节收纳架构评审中的核心挑战与对策，确保方案在工程落地时可应对物理/数学矛盾。

### 11.1 传播速度与帧率的脱节（最严重）

- **矛盾**：电磁波速 \(c \approx 3 \times 10^8\) m/s；10 m 微格、60 FPS（约 16 ms/帧）时，一帧内真实波已飞行约 **4 800 km**。若“每帧向 6 邻扩散一步”，波前仅前进 10 m/帧，等效波速约 600 m/s，相当于**极慢的粘性扩散**，与真实时延 \(R = (t_{rx}-t_{tx}) \cdot c/2\) 严重不符。
- **对策（三选一或组合）**：  
  - **波前索引（Wavefront Indexing）**：不按“每帧扩散一步”模拟波速，而是为每个能量包/脉冲维护“已传播距离”或“波前半径”。接收端采样时，直接按**距离**计算该脉冲是否已到达该格（\( \|\mathbf{x} - \mathbf{x}_{tx}\| \leq c \cdot (t - t_{tx}) \)），并据此决定该格是否参与该脉冲的回波贡献；传播步只负责**反射/绕射**（能量在障碍周围的再分布），不负责“波跑多远”。  
  - **子步迭代（Sub-stepping）**：每帧内执行多次传播子步，使“等效波速”接近 c；子步数 = \(c \cdot dt / \text{cellSize}\) 量级，例如 16 ms 内要跑 10 m 需约 1 步，但要跑 4 800 km 需约 3.2e5 步，不可行；故**子步仅适合短距离或粗格**，长距离仍建议波前索引。  
  - **解析式距离场**：发射时写入“脉冲源位置 + 发射时间”；采样时不做逐步扩散，而是对每个格按 \(R = \|\mathbf{x} - \mathbf{x}_{tx}\|\) 与 \(t_{rx} - t_{tx}\) 判断是否在“光锥”内，并直接用 \(R = (t_{rx}-t_{tx}) \cdot c/2\) 算回波距离；遮挡与反射则用占位网格做“射线–体素”或“可见性掩码”修正。  
- **推荐**：**波前索引 + 传播只做反射/多径**。即：直射路径的到达时间由几何距离与 c 解析得到；传播循环只更新“遇到固体后的反射/绕射能量分布”，这样帧率与波速脱钩问题从根上消除，且 \(t_{tx}\) 需**微秒级精度**（浮点时间戳），避免帧级离散导致距离 bin 抖动。

### 11.2 10 m 体素的精度缺失（采样走样）

- **问题**：10 m 格导致原始距离门约 10 m，战术雷达常需 1 m 级甚至更细；远距离 10 m 格张角很小，角度分辨率也受限于格尺寸。
- **对策**：见下节 **§12 亚体素精度**：在格内存**相干元数据**（精细距离偏移或相位），用插值实现亚体素距离/角度精度。

### 11.3 反射逻辑与法线质量

- **问题**：6 邻反射基于曼哈顿邻居，倾斜地形会产生锯齿式法线，回波方位易抖动。
- **对策**：固体格法线不只用“高度差梯度”，可引入**占位体的距离场梯度（SDF Gradient）**：对占位网格做平滑或 SDF 拟合，再在格心取梯度作为法线，使反射方向连续、减少体素锯齿感。

### 11.4 架构优化建议汇总

| 模块 | 改进建议 |
|------|----------|
| **存储层** | 稀疏体素八叉树（SVO）：波束通常为扇形，不必为全图 200 m 块满分配。 |
| **发射层** | “能量包（Energy Packets）”带 TTL：在体素间跳跃，遇固体分裂/转向，便于与波前索引结合。 |
| **多普勒** | 体素能量附带 `SourceVelocity` 或频率标记；遇移动实体格时按相对速度修正 FreqMask/多普勒频移。 |
| **时间戳** | \(t_{tx}\) 使用微秒级精度（浮点秒），保证 \(R = (t_{rx}-t_{tx}) \cdot c/2\) 可达 1 m 级距离分辨率。 |

## 12. 亚体素精度：10 m 格下实现约 1 m 级距离精度

在保持 10 m 体素网格的前提下，通过**相干元数据 + 插值**在检测阶段得到约 1 m 级距离分辨率与更平滑的方位。

### 12.1 思路概述

- **格内不只有功率**：为每个有能量的格增加**距离相关元数据**，使“距离”在格内连续可插值，而不是整格一个常数。
- **两种等价表述**：  
  - **精细距离偏移**：存“该格对某脉冲的等效回波距离”\(R_{\text{cell}}\)（或“距离残差”\(\delta R = R - R_{\text{cell}}^{\text{整数}}\)），接收时用邻格插值得到亚格精度的 \(R\)。  
  - **相位/双程相位**：存双程相位 \(\phi = 2\pi \cdot 2R/\lambda\)，利用相位连续可插值，反解 \(R = \phi \lambda/(4\pi)\)；需波长 \(\lambda\) 与相位解缠。

下面以**距离残差 + 三线性插值**为主方案（实现简单、不涉及相位解缠）。

### 12.2 格内距离元数据（相干元数据）

- 对每个“有能量”的微格，在写入或传播时维护：  
  - **功率** \(P\)（已有）；  
  - **主方向** \(\mathbf{d}\)（已有）；  
  - **到发射机的几何距离** \(R_{\text{geom}}\)：\(R_{\text{geom}} = \|\mathbf{x}_{\text{cell}} - \mathbf{x}_{tx}\|\)，其中 \(\mathbf{x}_{\text{cell}}\) 为格心或反射点；  
  - 或直接存**双程时延** \(\tau = 2 R_{\text{geom}}/c\)，接收时 \(R = c \cdot \tau / 2\)。  
- 反射格：\(R_{\text{geom}}\) 取“发射机 → 反射点 → 接收机”的双程路径长，即 \(R_{\text{geom}} = \|\mathbf{x}_{\text{refl}} - \mathbf{x}_{tx}\| + \|\mathbf{x}_{\text{refl}} - \mathbf{x}_{rx}\|\)；若单站则 \(R_{\text{geom}} = 2 \|\mathbf{x}_{\text{refl}} - \mathbf{x}_{tx}\|\)。

这样每个格带有“该格代表的回波距离”，分辨率不再被格尺寸硬限。

### 12.3 接收端：三线性插值得到亚格距离

- 接收机位置 \(\mathbf{x}_{rx}\) 落在某格内或跨格；取该点所在格的** 8 邻格**（与现有 GetPowerAt 的三线性插值一致），每个邻格提供 \((P_k, R_k)\)，\(k=1..8\)。  
- **功率加权距离**（稳健）：  
  \[
  R_{\text{out}} = \frac{\sum_k w_k P_k R_k}{\sum_k w_k P_k}, \qquad w_k = \text{三线性权重}(k).
  \]  
  这样 \(R_{\text{out}}\) 连续且偏向强回波格，可得亚 10 m 的 \(R\)。  
- **或相位加权**：若存相位 \(\phi_k = 4\pi R_k/\lambda\)，先对 \(\phi_k\) 做三线性插值得 \(\bar{\phi}\)，再 \(R_{\text{out}} = \bar{\phi} \lambda/(4\pi)\)；需同一脉冲、同频，且注意 \(2\pi\) 模糊（短脉宽/高 PRF 可限制不模糊距离）。

### 12.4 距离 bin 与 1 m 级分辨率

- 检测阶段：**距离 bin 宽度**设为 1 m（或 0.5 m），与体素 10 m 解耦。  
- 每个 bin 的功率由**所有对该 bin 有贡献的格**聚合：格 k 的 \(R_k\) 落在 \([R_{\text{bin}}-0.5, R_{\text{bin}}+0.5]\) 则加入该 bin；或用平滑核（如三角窗）按 \(R_k\) 与 bin 中心的距离加权。  
- 这样**原始采样仍是 10 m 格**，但**检测输出**是 1 m 级距离门；亚格精度来自格内 \(R_k\) 的差异与插值，而不是把体素改成 1 m（避免内存与传播爆炸）。

### 12.5 角度与方位

- **主方向** \(\mathbf{d}\) 已有；接收端用 \(\mathbf{d}\) 或“接收机→格心”方向得到方位/俯仰。  
- 亚格角度：对邻格的 \(\mathbf{d}_k\) 做三线性或功率加权平均，再归一化，可减轻单格方向量化带来的方位抖动。

### 12.6 小结

- **10 m 体素 + 格内 \(R_{\text{geom}}\)（或 \(\tau\)）+ 功率加权三线性插值 + 1 m 距离 bin**，可在不缩小体素的前提下，在检测端得到约 **1 m 级距离分辨率** 和更平滑的方位。  
- 若需更高精度，可再引入相位与解缠；若仅需战术级 1 m，上述“距离元数据 + 插值”即可。

## 13. 固体材质如何获取（用于反射率与 RCS）

波方案中固体格的**反射系数**（以及若输出 RDF_RadarSample 时的 `m_MaterialType` / `m_ReflectionCoefficient`）需要材质信息。当前射线方案通过 `TraceParam.SurfaceProps`（GameMaterial）在命中点取得材质；体素占位**无命中点**，需另辟来源。以下方案在**零扫描射线**前提下尽量复用现有 `RDF_RCSModel.GetMaterialReflectivity(matName)`。

### 13.1 需求与约束

- **用途**：传播时固体格的反射系数 \(\rho\)（0～1）；检测点若需填 `m_MaterialType`，可从占位格读出。
- **约束**：占位填充阶段可接受**有限次**查询（如每格一次、且可缓存），**检测/传播主路径**不再为“每条扫描射线”做 Trace。

### 13.2 地形格材质

- **方案 A（推荐：按需单次 Trace，缓存）**  
  - 在**占位填充**该块时，对每个判为“地形”的微格，从格心**正上方**（例如格心 + (0, 5, 0)）向格心做**一次短 Trace**，取 `SurfaceProps` → `GetName()` 得到材质名字符串，写入占位层的 `m_MaterialType[]` 或直接存 `reflectivity`。  
  - 每个地形格**只 Trace 一次**，结果随占位块缓存；块未失效前不再查询。射线量 = 该块内地形格数（最多 8000/块），与扫描频率、射线数无关。  
- **方案 B（零射线，统一默认）**  
  - 所有地形格不查材质，统一用默认 `"ground"`（或 `"dirt"`），对应现有 `GetMaterialReflectivity("ground")` ≈ 0.1。实现简单，适合先跑通再细化。  
- **方案 C（引擎地形材质 API，若存在）**  
  - 若 Enfusion 提供“给定 (x, z) 或世界坐标返回地表材质/表面类型”的 API（如 `GetSurfaceType(x,z)`、`GetTerrainMaterial` 等），则地形格可直接用该 API 取材质名，零射线。需查引擎文档确认。

**建议**：先实现 B，再在需要更细反射差异时上 A 或 C；占位块懒加载时对地形格做一次 Trace 并缓存，成本可控。

### 13.3 实体格材质

- **方案 A（实体类型 → 材质表，零射线）**  
  - 占位填充时已知“该格属于哪个实体”（已做 AABB 包含）。用**实体类型/类名**查表得到材质名，再调用现有 `GetMaterialReflectivity(matName)`。  
  - 例如：`SCR_BaseVehicle` / `CarScript` → `"metal"`，`SCR_Character` → `"fabric"` 或 `"body"`，`SCR_Building` → `"concrete"`，未知 → `"metal"`。  
  - 不依赖 `IEntity.GetMaterial()`（引擎可能不提供），仅用现有类名/类型字符串即可，与现有 RCS 材质表一致。  
- **方案 B（脚本挂载材质标签）**  
  - 允许在实体上挂组件或扩展键，如 `m_RadarMaterialName = "metal"`；占位时若存在则优先用该字符串，否则回退到类型表。便于任务制作者微调个别载具/建筑。  
- **方案 C（按格单次 Trace，可选）**  
  - 对实体格从格心向实体内部或表面做一次短 Trace 取 `SurfaceProps`。仅在对单实体多材质（如车辆不同部位）有需求时考虑；成本 = 实体格数，通常高于类型表。

**建议**：默认用 A（类型→材质表），可选 B；C 仅在确有需求且性能允许时启用。

### 13.4 占位层数据结构扩展

- 在占位数组旁增加（可按块懒分配）：  
  - **m_MaterialName[]**：每格一个 string（或 hash），地形/实体格写入材质名，空气格可空或 `""`。  
  - 或 **m_Reflectivity[]**：每格一个 float，填充时调用 `GetMaterialReflectivity(materialName)` 写入，传播时直接用，避免每帧查表。  
- 传播时：若邻居为固体，读该格的 `m_Reflectivity`（或现场查 `GetMaterialReflectivity(m_MaterialName[idx])`），反射能量乘以该系数。

### 13.5 与现有 RDF 的对接

- 反射率继续使用 **`RDF_RCSModel.GetMaterialReflectivity(string)`**，材质名字符串与现有射线方案一致（如 `"metal_01"`, `"grass"`, `"ground"`），便于共用同一套 RCS/反射表。  
- 检测点若需填 `RDF_RadarSample.m_MaterialType`，从检测点所在格（或插值主贡献格）的占位材质字段读取即可。

### 13.6 小结

| 固体类型 | 推荐方案 | 射线使用 |
|----------|----------|----------|
| 地形 | 默认 "ground"；或占位时每地形格一次向下 Trace 并缓存 | 0 或 每地形格 1 次（一次性） |
| 实体 | 实体类型/类名 → 材质表；可选脚本材质标签 | 0 |
| 可选 | 实体格单次 Trace 取 SurfaceProps | 每实体格 1 次（可选） |

这样在**不增加扫描射线**的前提下，固体反射能区分金属/混凝土/地面/植被等，并与现有材质反射率逻辑一致。

### 13.7 引擎 GameMaterial 与 BallisticInfo（Enfusion）

Enfusion 中 **GameMaterial**（即 `TraceParam.SurfaceProps` 命中后得到的类型）继承自 GameLibMaterial，并暴露：

- `GetVehicleInfo()` → ContactSurface  
- `GetSoundInfo()` → SoundInfo  
- `GetHitEffectInfo()` → HitEffectInfo  
- **`GetBallisticInfo()` → BallisticInfo**  
- `GetParticleEffectInfo()` → ParticleEffectInfo  

**.gamemat** 资源中可配置 BallisticInfo（密度、穿透抗性、厚度、Allow Deflection 等），与编辑器中 **.physmat** 的 Ballistic Info 对应。因此：

- **射线方案**：命中后已有 `sample.m_Surface`（GameMaterial），除 `GetName()` 查表外，可调用 **`surface.GetBallisticInfo()`**，若返回非 null，则用 BallisticInfo 的**数值**（如密度、Kinetic Penetration Resistance）映射为反射率，与材质名查表二选一或混合（优先 BallisticInfo，缺省再回退 GetName + GetMaterialReflectivity）。
- **波方案占位**：地形格做“单次 Trace 取材质”时，Trace 结果中的 SurfaceProps 同样可 `GetBallisticInfo()`，将得到的反射率写入占位格的 `m_Reflectivity[]`，与 13.2/13.4 一致。

**BallisticInfo 脚本 API**（Enfusion 生成）提供：`GetDensity()`、`GetThickness()`、`GetThicknessMax()`、`AllowsDeflection()`、`IsWaterSurface()`。当前 **RDF_RCSModel.GetReflectivityFromGameMaterial(GameMaterial, string)** 已实现：有 BallisticInfo 时用 `IsWaterSurface()`→0.6，否则用 `GetDensity()` 线性映射到 [0.05, 1.0]（密度 8 约等于 1）；无 BallisticInfo 时回退到材质名查表。波方案占位填充时若对地形格做单次 Trace，对得到的 GameMaterial 可同样调用该函数并缓存反射率。
